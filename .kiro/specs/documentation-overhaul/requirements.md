# Requirements Document — Documentation Overhaul

## Introduction

This document defines the requirements for a complete overhaul of the AI Support Workflow project documentation. The goal is to make the main README a rich and navigable entry point, verify coherence between documentation and current code, document all API endpoints with test examples via `.http` files, and create a navigation structure between documentation pages in the `docs/` folder.

## Glossary

- **Main_README**: The `README.md` file at the project root, serving as the documentation entry point.
- **Docs_Folder**: The `docs/` folder containing in-depth architectural documents.
- **In_Depth_Document**: A Markdown file in the `docs/` folder that covers a specific aspect of the project in detail.
- **HTTP_File**: The `AiSupportWorkflow.Presentation.http` file used to test API endpoints via IDE-integrated HTTP clients.
- **Architectural_Diagram**: A textual diagram (Mermaid or ASCII) representing the high-level structure of the system.
- **Navigable_Section**: A section of the Main_README that provides a summary of a topic and contains a link to the corresponding In_Depth_Document.
- **Navigation_Index**: A block of internal links present in each In_Depth_Document that allows navigation between documentation pages.
- **API_Endpoint**: An HTTP route exposed by the Presentation layer of the project (e.g., `POST /api/support/emails`).

## Requirements

### Requirement 1: Main README Structure

**User Story:** As a developer discovering the project, I want to find all essential information organized in clear sections in the main README, so that I can quickly understand what the project does, how it is built, and how to use it.

#### Acceptance Criteria

1. THE Main_README SHALL contain a project description section explaining that it is a spec-driven development experiment with Kiro, entirely generated by AI.
2. THE Main_README SHALL contain a "What It Does" section describing the complete system workflow: email reception, LLM classification, team routing, root cause analysis, and code fix generation.
3. THE Main_README SHALL contain a "Technologies" section listing the main technologies used (.NET 10, Akka.NET, Semantic Kernel, OpenAI) with links to their official documentation.
4. THE Main_README SHALL contain a "Getting Started" section with the steps to clone, configure the API key, and start the project.
5. THE Main_README SHALL contain an "API Endpoints" section listing all available endpoints with HTTP method, route, description, and a reference to the HTTP_File for testing.
6. THE Main_README SHALL contain an "Architecture" section with an Architectural_Diagram showing the four layers (Domain, Application, Infrastructure, Presentation) and the dependency flow.
7. THE Main_README SHALL contain in-depth sections (Navigable_Section) for Clean Architecture, Actor Architecture, and Semantic Kernel Integration, each with a summary and a link to the corresponding In_Depth_Document in `docs/`.

### Requirement 2: API Endpoint Documentation

**User Story:** As a developer who wants to test the system, I want to find clear documentation for each endpoint with concrete request examples, so that I can use the `.http` file to verify functionality.

#### Acceptance Criteria

1. THE Main_README SHALL document the `POST /api/support/emails` endpoint specifying the expected JSON body (fields `Sender`, `Subject`, `Body`), success and error responses, and a request example.
2. THE Main_README SHALL document the `GET /api/support/issues/{id}` endpoint specifying the `id` parameter (GUID), the expected response (workflow state), and a request example.
3. THE Main_README SHALL document the `GET /api/support/issues` endpoint specifying that it returns the list of all processed issues.
4. THE Main_README SHALL document the `GET /api/support/stream` endpoint specifying that it is an SSE stream of real-time updates and that it requires visualization to be enabled in the configuration.
5. THE Main_README SHALL document the `GET /api/support/agents` endpoint specifying that it returns the current state of all AI agents and that it requires visualization to be enabled.
6. THE Main_README SHALL contain an explicit reference to the HTTP_File path (`src/AiSupportWorkflow.Presentation/AiSupportWorkflow.Presentation.http`) with instructions on how to use it to test the endpoints.

### Requirement 3: Architectural Diagram

**User Story:** As a developer, I want to see a visual schema of the project architecture in the README, so that I can understand at a glance the layer structure and the data flow.

#### Acceptance Criteria

1. THE Main_README SHALL contain a textual diagram (Mermaid or ASCII art format) representing the four Clean Architecture layers and their dependencies.
2. THE Main_README SHALL contain a textual diagram representing the workflow pipeline flow: email reception → classification → team routing → agent assignment → resolution → code change generation.
3. WHEN the architectural diagram is rendered, THE Main_README SHALL clearly show the direction of dependencies (from Presentation toward Domain).

### Requirement 4: Document Navigation

**User Story:** As a developer consulting the documentation, I want to easily navigate between the main README and the in-depth documents, so that I do not lose context while reading.

#### Acceptance Criteria

1. THE Main_README SHALL contain direct links to each In_Depth_Document in the `docs/` folder (clean-architecture.md, actor-architecture.md, semantic-kernel-integration.md).
2. WHEN an In_Depth_Document is opened, THE In_Depth_Document SHALL contain a Navigation_Index at the top of the document with links to return to the Main_README and to reach the other in-depth documents.
3. THE Navigation_Index SHALL use a consistent format across all in-depth documents.
4. THE Docs_Folder SHALL contain an `index.md` file that serves as a documentation map, listing all available documents with a brief description and direct links.

### Requirement 5: Documentation-Code Coherence

**User Story:** As a developer, I want the documentation to faithfully reflect the current state of the code, so that I am not misled by outdated information.

#### Acceptance Criteria

1. THE Main_README SHALL list API endpoints that correspond exactly to the routes defined in the `SupportEmailEndpoints.cs`, `WorkflowStatusEndpoints.cs`, and `VisualizationEndpoints.cs` files.
2. THE Main_README SHALL describe the technologies with versions that correspond to those specified in the project's `.csproj` files.
3. WHEN an In_Depth_Document describes a code structure (classes, interfaces, namespaces), THE In_Depth_Document SHALL reflect the names and structures actually present in the source code.
4. THE Main_README SHALL describe the project folder structure in a way that is coherent with the actual repository layout.
5. THE Main_README SHALL describe the build and test commands (`dotnet build`, `dotnet test`, `dotnet run`) with the correct project paths.

### Requirement 6: DummyApps and Test Scenarios Documentation

**User Story:** As a developer who wants to understand how to test the system end-to-end, I want to find an explanation of the DummyApps and predefined bug scenarios, so that I know which test emails to send.

#### Acceptance Criteria

1. THE Main_README SHALL contain a section describing the purpose of the DummyApps (ApplicationA and ApplicationB) as test fixtures.
2. THE Main_README SHALL explain that each application has predefined bug scenarios (BackendBug, FrontendBug, QualityTestIssue) documented in their respective `BugScenarios.md` files.
3. THE Main_README SHALL indicate that the HTTP_File contains ready-made requests for testing each scenario, including edge cases (out-of-scope, ambiguous routing, failed routing, empty input).

### Requirement 7: Testing and Quality Section

**User Story:** As a developer contributing to the project, I want to know how the tests are organized and how to run them, so that I can verify my changes.

#### Acceptance Criteria

1. THE Main_README SHALL contain a "Testing" section describing the test organization: unit tests with xUnit and NSubstitute, property-based tests with FsCheck.
2. THE Main_README SHALL list the commands to run the tests: all tests (`dotnet test AiSupportWorkflow.sln`), unit tests only (`dotnet test tests/AiSupportWorkflow.UnitTests`), property tests only (`dotnet test tests/AiSupportWorkflow.PropertyTests`).
3. THE Main_README SHALL mention the adopted testing conventions: one test class per service, Arrange-Act-Assert pattern, NSubstitute for mocking.
